Introduction

This guide is designed to help network administrators and students understand common basic network problems and how to simulate and resolve them step-by-step in a Cisco Packet Tracer environment. Each problem is categorized by its respective network layer, and the solution steps are explained in detail. This will allow you to reinforce your theoretical knowledge with practical applications.

Problem 1.1: Communication Issue Due to Incorrect VLAN Assignment

Concept:

VLAN (Virtual Local Area Network) allows for the creation of multiple logical broadcast domains on a single physical switch, logically segmenting networks. Devices in different VLANs cannot communicate directly (without a Layer 3 device), even if they are in the same IP subnet.

Problem Description:

Two computers (PC0 and PC1) are in the same IP address block (e.g., 192.168.1.0/24) but cannot ping each other. Upon investigation, it's found that one of the PCs is mistakenly assigned to a different VLAN, or both are assigned to VLANs different from what they should be.

Packet Tracer Scenario:

Required Devices:

  • 1 x Switch (e.g., 2960)
  • 2 x PCs.

Connections:

  • PC0 -> Switch Fa0/1
  • PC1 -> Switch Fa0/2

Basic Configurations:

PC0 IP: 192.168.1.10 / Subnet Mask: 255.255.255.0

PC1 IP: 192.168.1.11 / Subnet Mask: 255.255.255.0

Create VLAN 10 (DATA_VLAN) and VLAN 20 (GUEST_VLAN) on the switch.

Switch> enable
Switch# configure terminal
Switch(config)# vlan 10
Switch(config-vlan)# name DATA_VLAN
Switch(config-vlan)# exit
Switch(config)# vlan 20
Switch(config-vlan)# name GUEST_VLAN
Switch(config-vlan)# exit

Simulating the Problem:

Assign PC0's connected port (Fa0/1) to VLAN 10.

Switch(config)# interface FastEthernet0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 10
Switch(config-if)# exit

Assign PC1's connected port (Fa0/2) incorrectly to VLAN 20.

Switch(config)# interface FastEthernet0/2
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 20
Switch(config-if)# exit

Try to ping PC1 from PC0 (Command Prompt: ping 192.168.1.11). The ping will fail.

PT Visual: PC0 Command Prompt screen shows "Request timed out." messages.

Check VLAN assignments on the switch:

Switch# show vlan brief

PT Visual: The output of show vlan brief shows Fa0/1 in VLAN 10 and Fa0/2 in VLAN 20.

Solution Steps:

Assign PC1's connected port Fa0/2 to the correct VLAN (VLAN 10).

Switch(config)# interface FastEthernet0/2
Switch(config-if)# switchport access vlan 10
Switch(config-if)# exit

Verifying the Solution:

Try to ping PC1 from PC0 again (ping 192.168.1.11). The ping will be successful.

PT Visual: PC0 Command Prompt screen shows successful ping replies like "Reply from 192.168.1.11: bytes=32 time<1ms TTL=128".

Check VLAN assignments on the switch again:

Switch# show vlan brief

PT Visual: The output of show vlan brief shows both Fa0/1 and Fa0/2 in VLAN 10.

Problem 1.2: Port Shutdown Due to Port Security Violation

Concept:

Port Security is a security feature that allows only specific MAC addresses or a specific number of MAC addresses to connect to a switch port. In case of a violation, the port can be shut down (err-disabled), restricted, or the violation can be logged (protect).

Problem Description:

Port Security is configured on a switch port. When the allowed number of MAC addresses is exceeded or an unauthorized MAC address connects, the port goes into "err-disabled" (shutdown) mode, and communication through that port is cut off.

Packet Tracer Scenario:

Required Devices:

  • 1 x Switch (e.g., 2960)
  • 2 x PCs (PC-Authorized, PC-Unauthorized).

Connections:

Initially PC-Authorized -> Switch Fa0/1. Later, PC-Unauthorized -> Switch Fa0/1.

Basic Configurations:

PCs can be given any IP address (e.g., 192.168.1.x/24). The goal is to test L2 connectivity.

Simulating the Problem:

Enable Port Security on switch port Fa0/1, allow a maximum of 1 MAC address, and set the violation mode to shutdown. Configure it to learn MAC addresses dynamically (sticky).

Switch> enable
Switch# configure terminal
Switch(config)# interface FastEthernet0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport port-security
Switch(config-if)# switchport port-security maximum 1
Switch(config-if)# switchport port-security mac-address sticky
Switch(config-if)# switchport port-security violation shutdown
Switch(config-if)# exit

Connect PC-Authorized to Fa0/1. Ping from PC-Authorized to the switch (if it has an SVI IP) or another device on the network to make the port learn its MAC address.

PT Visual: The link light on Switch Fa0/1 port turns green.

Check the port security status:

Switch# show port-security interface FastEthernet0/1

PT Visual: The output shows "Port Security: Enabled", "Port Status: Secure-up", "Violation Mode: Shutdown", "Sticky MAC Addresses: 1", and the learned MAC address.

Now, disconnect PC-Authorized from Fa0/1 and connect PC-Unauthorized to the same port (Fa0/1).

When you try to send a ping from PC-Unauthorized, or after a short while for the switch to update its ARP table, the port will shut down due to a violation.

PT Visual: The link light on Switch Fa0/1 port turns red (or orange then red).

Check the port status:

Switch# show interfaces FastEthernet0/1 status

PT Visual: The output shows the port is "err-disabled".

Switch# show port-security interface FastEthernet0/1

PT Visual: The output shows "Port Status: Secure-shutdown", "SecurityViolationCount: 1".

Solution Steps:

First, understand the cause of the violation. In this scenario, an unauthorized MAC address was connected.

To reactivate the port, first shut it down and then bring it up:

Switch(config)# interface FastEthernet0/1
Switch(config-if)# shutdown
Switch(config-if)# no shutdown
Switch(config-if)# exit

Important Note: Simply re-enabling the port is not a permanent solution. You must either review the Port Security policy (e.g., increase the maximum MAC count, statically add the new MAC address) or ensure only the authorized device is connected. In this scenario, disconnecting PC-Unauthorized and reconnecting PC-Authorized resolves the immediate issue.

Verifying the Solution:

Reconnect PC-Authorized to Fa0/1.

Observe that the link light turns green.

PT Visual: The link light on Switch Fa0/1 port turns green.

Perform a ping test from PC-Authorized. It will be successful.

Check the port status:

Switch# show interfaces FastEthernet0/1 status

PT Visual: The output shows the port is "connected" and "up".

Problem 1.3: Inter-VLAN Communication Issue (Without Router-on-a-Stick) Due to Missing or Incorrect Trunk Configuration

Concept:

Trunk ports are used to carry traffic for multiple VLANs (usually using IEEE 802.1Q tagging) between two switches. If multiple VLANs need to communicate across two switches, the link between them must be configured as a trunk.

Problem Description:

Two switches (SW1 and SW2) are connected. The same VLANs (e.g., VLAN 10, VLAN 20) are defined on both switches. A PC in VLAN 10 on SW1 cannot communicate with another PC in VLAN 10 on SW2. The reason is that the inter-switch link is not configured as a trunk or is misconfigured.

Packet Tracer Scenario:

Required Devices:

  • 2 x Switches (e.g., 2960)
  • 4 x PCs.

Connections:

  • PC0 (VLAN 10) -> SW1 Fa0/1
  • PC1 (VLAN 20) -> SW1 Fa0/2
  • SW1 Gi0/1 -> SW2 Gi0/1 (Inter-switch link)
  • PC2 (VLAN 10) -> SW2 Fa0/1
  • PC3 (VLAN 20) -> SW2 Fa0/2

Basic Configurations:

All PCs can be in the same IP subnet (e.g., 192.168.1.0/24) as the goal is to test communication within the same VLAN across switches.

PC0: 192.168.1.10/24

PC1: 192.168.1.11/24

PC2: 192.168.1.12/24

PC3: 192.168.1.13/24

Create VLAN 10 and VLAN 20 on SW1 and SW2.

// Same commands on SW1 and SW2
Switch> enable
Switch# configure terminal
Switch(config)# vlan 10
Switch(config-vlan)# name VLAN10_DATA
Switch(config-vlan)# exit
Switch(config)# vlan 20
Switch(config-vlan)# name VLAN20_VOICE
Switch(config-vlan)# exit

Assign ports to their respective VLANs:

// On SW1
Switch(config)# interface FastEthernet0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 10
Switch(config-if)# exit
Switch(config)# interface FastEthernet0/2
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 20
Switch(config-if)# exit

// On SW2
Switch(config)# interface FastEthernet0/1
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 10
Switch(config-if)# exit
Switch(config)# interface FastEthernet0/2
Switch(config-if)# switchport mode access
Switch(config-if)# switchport access vlan 20
Switch(config-if)# exit

Simulating the Problem:

Do not configure the inter-switch link ports (Gi0/1) as trunks (they might be in access mode by default or auto mode via DTP, which can cause issues).

Try to ping PC2 (VLAN 10, SW2) from PC0 (VLAN 10, SW1) (ping 192.168.1.12). The ping will fail.

PT Visual: PC0 Command Prompt screen shows "Request timed out." messages.

Similarly, a ping from PC1 (VLAN 20, SW1) to PC3 (VLAN 20, SW2) will also fail.

Solution Steps:

Configure the inter-switch link ports (Gi0/1) on SW1 and SW2 as trunk ports.

// On SW1
Switch(config)# interface GigabitEthernet0/1
Switch(config-if)# switchport mode trunk
Switch(config-if)# exit

// On SW2
Switch(config)# interface GigabitEthernet0/1
Switch(config-if)# switchport mode trunk
Switch(config-if)# exit

(Optional but good practice) You can specify allowed VLANs and the native VLAN:

// On SW1 and SW2 on interface Gi0/1
Switch(config-if)# switchport trunk allowed vlan 10,20
Switch(config-if)# switchport trunk native vlan 99 // (If used, VLAN 99 must be created on both switches)

Verifying the Solution:

Try to ping PC2 (VLAN 10, SW2) from PC0 (VLAN 10, SW1) again (ping 192.168.1.12). The ping will be successful.

PT Visual: PC0 Command Prompt screen shows successful ping replies.

Try to ping PC3 (VLAN 20, SW2) from PC1 (VLAN 20, SW1) (ping 192.168.1.13). The ping will be successful.

Check the trunk port status on the switches:

Switch# show interfaces trunk

PT Visual: The output of show interfaces trunk lists port Gi0/1 with "Mode: on", "Encapsulation: 802.1q", "Status: trunking", and "Vlans allowed on trunk: 10,20".

Problem 2.1: Inability to Access Different Networks Due to Incorrect Default Gateway Configuration

Concept:

The Default Gateway is the IP address of the router interface that a device uses to send packets to a destination outside its own local network. If this address is incorrect or not defined, the device cannot communicate outside its local network.

Problem Description:

A PC (e.g., PC0) can communicate with other devices in its local network (e.g., another PC connected to the same switch) but cannot reach a destination in a different network (e.g., the internet, a server in another VLAN).

Packet Tracer Scenario:

Required Devices:

  • 1 x PC
  • 1 x Switch (e.g., 2960)
  • 1 x Router (e.g., 4331)
  • A Server or another PC can be added to simulate a remote network.

Connections:

  • PC0 -> Switch Fa0/1
  • Switch Gi0/1 -> Router G0/0/0 (Local network connection)
  • Router G0/0/1 -> Server0 (Remote network connection, representing the internet)

Basic Configurations:

PC0:

  • IP Address: 192.168.1.10
  • Subnet Mask: 255.255.255.0
  • Default Gateway: (This is where the problem will be introduced)

Router:

  • Interface G0/0/0 (facing local network): IP Address: 192.168.1.1 / Subnet Mask: 255.255.255.0, no shutdown
  • Interface G0/0/1 (facing remote network): IP Address: 203.0.113.1 / Subnet Mask: 255.255.255.0, no shutdown

Server0 (Remote Network):

  • IP Address: 203.0.113.100
  • Subnet Mask: 255.255.255.0
  • Default Gateway: 203.0.113.1 (Router's interface facing that network)

Simulating the Problem:

In PC0's IP configuration, enter an incorrect Default Gateway (e.g., 192.168.1.254) or leave it blank.

PT Visual: PC0's IP Configuration window shows the Default Gateway field incorrect or empty.

Try to ping PC0's supposed Default Gateway address (Router G0/0/0: 192.168.1.1). This will be successful (as they are in the same L2 segment).

PT Visual: ping 192.168.1.1 from PC0 Command Prompt is successful.

Try to ping Server0 (203.0.113.100) in the remote network from PC0. The ping will fail (you'll get "Request timed out" or "Destination host unreachable").

PT Visual: ping 203.0.113.100 from PC0 Command Prompt fails.

Solution Steps:

Open PC0's IP configuration.

Update the Default Gateway address with the correct IP address of the router interface for PC0's network (192.168.1.1).

PT Visual: PC0's IP Configuration window shows the Default Gateway field corrected to 192.168.1.1.

Verifying the Solution:

Try to ping Server0 (203.0.113.100) in the remote network from PC0 again. The ping will be successful.

PT Visual: ping 203.0.113.100 from PC0 Command Prompt is successful.

You can run tracert 203.0.113.100 from PC0 to see the path the packet takes (it should show 192.168.1.1 as the first hop).

PT Visual: The tracert output shows 192.168.1.1 as the first line.

Problem 2.2: Inability to Reach Destination Network Due to Missing Static Route

Concept:

Routers use routing tables to decide how to forward packets to their destinations. When a router needs to send a packet to a network it's not directly connected to, it must know how to reach that network. This information is provided by dynamic routing protocols (RIP, EIGRP, OSPF, etc.) or by manually adding static routes.

Problem Description:

A router (R1) cannot forward packets to a remote network (e.g., a network behind R2) because it has no entry for it in its routing table. Therefore, a PC behind R1 cannot reach a PC behind R2.

Packet Tracer Scenario:

Required Devices:

  • 2 x PCs
  • 2 x Routers (e.g., 4331)

Connections:

  • PC0 -> R1 G0/0/0
  • R1 G0/0/1 -> R2 G0/0/1 (Serial or Ethernet link between routers)
  • R2 G0/0/0 -> PC1

Basic Configurations:

PC0:

  • IP Address: 192.168.1.10/24
  • Default Gateway: 192.168.1.1

R1:

  • G0/0/0: IP 192.168.1.1/24, no shutdown
  • G0/0/1: IP 10.0.0.1/30, no shutdown

R2:

  • G0/0/1: IP 10.0.0.2/30, no shutdown
  • G0/0/0: IP 192.168.2.1/24, no shutdown

PC1:

  • IP Address: 192.168.2.10/24
  • Default Gateway: 192.168.2.1

Simulating the Problem:

Configure the IP addresses and interfaces as above. Do not add static routes to the routers yet.

Try to ping PC1 from PC0 (ping 192.168.2.10). The ping will fail.

PT Visual: PC0 Command Prompt shows "Request timed out." or "Destination host unreachable" messages from R1.

Check R1's routing table:

R1# show ip route

PT Visual: R1's show ip route output does not show an entry for the 192.168.2.0/24 network. Only directly connected networks (192.168.1.0/24 and 10.0.0.0/30) are listed.

Similarly, R2's routing table will not have an entry for the 192.168.1.0/24 network.

Solution Steps:

On R1, add a static route to reach the 192.168.2.0/24 network via R2's 10.0.0.2 address as the next-hop:

R1> enable
R1# configure terminal
R1(config)# ip route 192.168.2.0 255.255.255.0 10.0.0.2
R1(config)# end

On R2, add a static route to reach the 192.168.1.0/24 network via R1's 10.0.0.1 address as the next-hop (communication must be two-way):

R2> enable
R2# configure terminal
R2(config)# ip route 192.168.1.0 255.255.255.0 10.0.0.1
R2(config)# end

Verifying the Solution:

Try to ping PC1 from PC0 again (ping 192.168.2.10). The ping will be successful.

PT Visual: PC0 Command Prompt shows successful ping replies.

Check the routing tables of R1 and R2 again:

R1# show ip route
R2# show ip route

PT Visual: R1's show ip route output now shows a static route "S 192.168.2.0/24 [1/0] via 10.0.0.2". R2 will similarly show a route for 192.168.1.0/24.

Problem 2.3: Unwanted Access Blocked (or Desired Access Accidentally Blocked) by a Simple Standard ACL

Concept:

An Access Control List (ACL) is applied to router interfaces to filter traffic (permit/deny) coming from specific IP addresses or going to specific IP addresses. Standard ACLs filter based only on the source IP address.

Problem Description:

A standard ACL is used to block access from one network segment (source IP) to another (destination) or to permit only specific sources. However, if the ACL is written incorrectly (e.g., order of deny and permit statements, missing permit any statement), desired access can also be blocked, or unwanted access might be permitted.

Packet Tracer Scenario:

Required Devices:

  • 2 x PCs (PC-Source, PC-Destination/Server)
  • 1 x Router (e.g., 4331)

Connections:

  • PC-Source -> Router G0/0/0
  • Router G0/0/1 -> PC-Destination/Server

Basic Configurations:

PC-Source: IP 192.168.1.10/24, DG 192.168.1.1

Router:

  • G0/0/0: IP 192.168.1.1/24, no shutdown
  • G0/0/1: IP 192.168.2.1/24, no shutdown

PC-Destination/Server: IP 192.168.2.10/24, DG 192.168.2.1

Simulating the Problem (Desired Access Accidentally Blocked):

Our goal is to block access from the entire 192.168.1.0/24 network except PC-Source (192.168.1.10) to PC-Destination (192.168.2.10). Let's accidentally block PC-Source.

Create the following ACL on the router and apply it inbound on interface G0/0/0:

Router> enable
Router# configure terminal
Router(config)# access-list 10 deny host 192.168.1.10 // Accidentally deny first
Router(config)# access-list 10 permit any             // Then permit any
Router(config)# interface GigabitEthernet0/0/0
Router(config-if)# ip access-group 10 in
Router(config-if)# end

This ACL blocks 192.168.1.10 and permits everything else, the opposite of what might be intended if the goal was only to permit 192.168.1.10.

Try to ping PC-Destination (192.168.2.10) from PC-Source (192.168.1.10). The ping will fail (you might get "Destination host unreachable" or an "Administratively Prohibited" ICMP message).

PT Visual: PC-Source Command Prompt shows a failed ping, possibly with an ICMP error from R1.

Check ACL matches on the router:

Router# show access-lists 10

PT Visual: The output of show access-lists 10 shows an increasing match count for the deny host 192.168.1.10 line.

Solution Steps:

Correct the order and logic of the ACL statements. Traffic we want to permit should be specified with permit first, then what we want to deny with deny. Often, to avoid the implicit deny any at the end (if not blocking everything), a permit any (or more specific permits) is added.

To achieve the goal: Permit only 192.168.1.10, deny the rest of 192.168.1.0/24. (Standard ACLs don't specify destination, so it's source-based. Let's fix the "accidentally blocked desired access" scenario from the prompt).

Correction (to permit PC-Source, and deny everything else from that source network if G0/0/0 is only for 192.168.1.0/24):

Router(config)# interface GigabitEthernet0/0/0
Router(config-if)# no ip access-group 10 in // Remove from interface first
Router(config-if)# exit
Router(config)# no access-list 10          // Remove the old ACL
Router(config)# access-list 10 permit host 192.168.1.10
// If other traffic from this interface/network should also be permitted, add more permit lines or 'permit any'.
// If the goal is *only* 192.168.1.10 is allowed and all other traffic from G0/0/0 is denied:
// Router(config)# access-list 10 deny   any (optional, as there's an implicit deny any)
Router(config)# interface GigabitEthernet0/0/0
Router(config-if)# ip access-group 10 in
Router(config-if)# end

Verifying the Solution:

Try to ping PC-Destination (192.168.2.10) from PC-Source (192.168.1.10) again. The ping will be successful (based on the last corrected ACL above).

PT Visual: PC-Source Command Prompt shows a successful ping.

Check ACL matches on the router:

Router# show access-lists 10

PT Visual: The output of show access-lists 10 shows an increasing match count for the permit host 192.168.1.10 line.

If you had another PC in your network (e.g., 192.168.1.11) and the ACL was permit host 192.168.1.10 followed by an implicit (or explicit) deny any, a ping attempt from 192.168.1.11 would fail.

Problem 3.1: Incorrect DHCP Server Configuration (Distributing Wrong Default Gateway or DNS Information)

Concept:

DHCP (Dynamic Host Configuration Protocol) automatically assigns IP addresses, subnet masks, default gateway, and DNS server addresses to clients on a network. If the DHCP server distributes incorrect information, clients will have issues accessing the network or the internet.

Problem Description:

Client PCs are receiving IP addresses from the DHCP server but cannot access the internet or resolve domain names. The cause is that the default gateway or DNS server address distributed by the DHCP server is incorrect.

Packet Tracer Scenario:

Required Devices:

  • 1 x PT Server (for DHCP and DNS services)
  • 1 x Switch
  • 1 x PC (DHCP client)
  • 1 x Router (acting as Default Gateway)

Connections:

  • PC -> Switch
  • PT Server -> Switch
  • Router (e.g., G0/0/0 interface) -> Switch

Basic Configurations:

Router (Gateway):

  • G0/0/0: IP 192.168.1.1/24, no shutdown

PT Server:

  • Static IP: 192.168.1.254/24
  • Default Gateway: 192.168.1.1
  • DNS Server: 192.168.1.254 (itself)

PC: IP configuration set to DHCP.

Simulating the Problem:

On the PT Server, enable the DHCP service.

Create a new pool (e.g., ServerPool).

For this pool, enter an incorrect Default Gateway (e.g., 192.168.1.100 instead of 192.168.1.1).

Enter an incorrect DNS Server (e.g., 192.168.1.200 instead of 192.168.1.254).

Start IP Address: 192.168.1.10

Subnet Mask: 255.255.255.0

Maximum Number of Users: 50

Click "Save" and ensure the service is "On".

PT Visual: PT Server's DHCP service settings screen shows Default Gateway and DNS Server fields filled with incorrect values.

On the PC, go to Desktop > IP Configuration and select DHCP. The PC will obtain an IP address.

Check the IP information received by the PC (in the same window or via ipconfig /all in the command prompt). You will see that the Default Gateway and DNS Server addresses are the incorrect values you entered in DHCP.

PT Visual: PC's IP Configuration window or ipconfig output shows incorrect Gateway and DNS addresses.

Try to ping an internet address (if available) or access a website by name from the PC. It will fail.

Solution Steps:

Return to the DHCP service settings on the PT Server.

For ServerPool (or your created pool), correct the Default Gateway address to the proper 192.168.1.1.

Correct the DNS Server address to the proper 192.168.1.254 (or a valid DNS server).

Click "Save".

PT Visual: PT Server's DHCP service settings screen shows Default Gateway and DNS Server fields filled with correct values.

On the PC, go to IP Configuration, select "Static" first, then "DHCP" again to renew the IP address.

Verifying the Solution:

Check the new IP information received by the PC. You will see that the Default Gateway and DNS Server addresses are now correct.

PT Visual: PC's IP Configuration window or ipconfig output shows correct Gateway and DNS addresses.

Try to ping an internet address or access a website by name from the PC. It will be successful (if DNS is working correctly).

Problem 3.2: Clients in Different VLANs Unable to Obtain IP from DHCP Server (Missing IP Helper-Address)

Concept:

DHCP requests (DISCOVER, REQUEST) are sent as broadcasts. Routers, by default, do not forward broadcast traffic to different networks (VLANs). Therefore, if the DHCP server is in a different VLAN/subnet from the client, the client's broadcast DHCP request cannot reach the server. The ip helper-address command allows the router to intercept DHCP broadcasts arriving on a specific interface and forward them as unicasts to the specified DHCP server.

Problem Description:

The DHCP server is located in one VLAN (e.g., Server VLAN). PCs in a different user VLAN send DHCP requests, but these requests, being broadcasts, cannot cross the router to reach the DHCP server. As a result, PCs fail to obtain an IP address (they usually get an APIPA 169.254.x.x address).

Packet Tracer Scenario:

Required Devices:

  • 1 x PT Server (DHCP)
  • 1 x Router (for Router-on-a-Stick)
  • 1 x Switch
  • 1 x PC

Connections:

  • PC -> Switch Fa0/1 (Access port for User VLAN)
  • PT Server -> Switch Fa0/2 (Access port for Server VLAN)
  • Switch Gi0/1 -> Router G0/0/0 (Trunk port)

Basic Configurations:

Switch:

  • Create VLAN 10 (USER_VLAN), VLAN 20 (SERVER_VLAN).
  • Assign Fa0/1 to VLAN 10 (switchport access vlan 10).
  • Assign Fa0/2 to VLAN 20 (switchport access vlan 20).
  • Make Gi0/1 a trunk port (switchport mode trunk).

Router (Router-on-a-Stick):

  • interface G0/0/0.10 (Subinterface for VLAN 10): encapsulation dot1q 10, ip address 192.168.10.1 255.255.255.0, no shutdown
  • interface G0/0/0.20 (Subinterface for VLAN 20): encapsulation dot1q 20, ip address 192.168.20.1 255.255.255.0, no shutdown
  • Enable the physical interface: interface G0/0/0, no shutdown

PT Server (DHCP Server):

  • Static IP: 192.168.20.100/24 (In Server VLAN)
  • Default Gateway: 192.168.20.1 (Gateway for Server VLAN)

DHCP Service:

  • Pool Name: USER_POOL
  • Default Gateway: 192.168.10.1 (Gateway for User VLAN)
  • DNS Server: (A valid address, e.g., 192.168.20.100 if it also provides DNS)
  • Start IP Address: 192.168.10.10
  • Subnet Mask: 255.255.255.0
  • "Save" the pool and turn the service "On".

PC: In VLAN 10, IP configuration set to DHCP.

Simulating the Problem:

Configure everything as described, but do not use the ip helper-address command on the router.

On the PC, go to Desktop > IP Configuration and select DHCP. After a while, the PC will fail to obtain an IP address and will get an APIPA address (169.254.x.x).

PT Visual: PC's IP Configuration window shows an APIPA address (169.254.x.x) and may display a "DHCP request failed" message.

Solution Steps:

Connect to the router's CLI.

Enter the router interface (subinterface G0/0/0.10 in this scenario) connected to the User VLAN (VLAN 10).

Add the ip helper-address [DHCP_server_IP_address] command. The DHCP server's IP address is 192.168.20.100.

Router> enable
Router# configure terminal
Router(config)# interface GigabitEthernet0/0/0.10
Router(config-subif)# ip helper-address 192.168.20.100
Router(config-subif)# end

Verifying the Solution:

On the PC, go to IP Configuration, select "Static" first, then "DHCP" again to renew the IP address.

You will see that the PC now obtains the correct IP address (e.g., 192.168.10.10) from the DHCP server (from USER_POOL), along with the gateway (192.168.10.1) and other information.

PT Visual: PC's IP Configuration window shows an IP address in the 192.168.10.x range obtained via DHCP, with correct gateway and DNS information.

Problem 3.3: Domain Name Resolution Failure Due to Missing Record on DNS Server

Concept:

DNS (Domain Name System) is a hierarchical system that translates human-readable domain names (e.g., www.cisco.com) into machine-understandable IP addresses (e.g., 104.86.223.135). If a DNS server does not have a record for a specific domain name, or if it's incorrect, access to that domain name will fail.

Problem Description:

A user tries to access a website (e.g., www.example-pt.com) via a browser but receives a "Hostname unresolved" or "Page not found" error. If the IP address of that site is known and can be pinged, the problem is likely with DNS resolution.

Packet Tracer Scenario:

Required Devices:

  • 1 x PT Server (for DNS and HTTP services)
  • 1 x PC
  • 1 x Switch

Connections:

PC -> Switch, PT Server -> Switch.

Basic Configurations:

PT Server:

  • Static IP: 192.168.1.100/24
  • Default Gateway: (Can be left blank or router IP if no outbound access needed)
  • DNS Service: "On"
  • HTTP Service: "On" (index.html file can have a simple "Hello World" text)

PC:

  • Static IP: 192.168.1.10/24
  • Default Gateway: (Same as above)
  • DNS Server: 192.168.1.100 (PT Server's IP)

Simulating the Problem:

On the PT Server, go to the DNS service. Do not add an A record for www.example-pt.com.

PT Visual: PT Server's DNS service settings screen shows no record for www.example-pt.com.

Open PC's Desktop > Web Browser.

In the address bar, type http://www.example-pt.com and click "Go". You will receive a "Hostname unresolved" error.

PT Visual: PC's web browser shows a "Hostname unresolved" error.

Open PC's Command Prompt and run ping www.example-pt.com. You will get a "Ping request could not find host..." error.

PT Visual: PC Command Prompt shows the ping www.example-pt.com command failing, stating it cannot resolve the name.

However, if you run ping 192.168.1.100 (PT Server's IP), it will be successful. This indicates network connectivity is fine, but name resolution is not working.

PT Visual: ping 192.168.1.100 from PC Command Prompt is successful.

Solution Steps:

Go to the DNS service settings on the PT Server.

In the "Resource Records" section:

  • Name: www.example-pt.com
  • Type: A Record (default)
  • Address: 192.168.1.100 (PT Server's own IP, as the HTTP service is also running there)

Click "Add". Then "Save" (usually auto-saves, but ensure).

PT Visual: PT Server's DNS service settings screen now shows an A record: www.example-pt.com -> 192.168.1.100.

Verifying the Solution:

Reopen PC's Desktop > Web Browser.

In the address bar, type http://www.example-pt.com and click "Go". This time, the index.html content from the PT Server's HTTP service (e.g., "Hello World") will be displayed.

PT Visual: PC's web browser displays the content of www.example-pt.com (index.html from PT Server).

Open PC's Command Prompt and run ping www.example-pt.com. This time, you will see the name resolved to 192.168.1.100 and the ping will be successful.

PT Visual: ping www.example-pt.com from PC Command Prompt works, shows the IP address (192.168.1.100), and receives successful ping replies.

You can also use the nslookup www.example-pt.com command to see the result of the DNS query.

PT Visual: The nslookup output shows that 192.168.1.100 is returned for www.example-pt.com.

Problem 4.1: No Connectivity Due to Interface Being Administratively Down

Concept:

Interfaces on network devices (routers, switches) can be administratively shut down using the shutdown command. In this state, even if the physical connection is correct, no data transmission will occur through the interface.

Problem Description:

A PC is physically connected to a switch with the correct cable, or a router is connected to another router, but there is no communication. In Packet Tracer, the link lights for the respective port are red (or one might be red, the other green/orange).

Packet Tracer Scenario:

Required Devices:

  • 1 x PC
  • 1 x Switch (e.g., 2960). (Or 2 x Routers).

Connections:

PC -> Switch Fa0/1.

Basic Configurations:

Give the PC an IP address: 192.168.1.10/24.

You can give the Switch a management IP (SVI) (e.g., 192.168.1.1/24 for VLAN 1), but it's not mandatory for this problem.

Simulating the Problem:

Connect the PC to Switch's Fa0/1 port. Normally, the link lights turn green.

PT Visual: The link lights on both ends of the connection between PC and Switch Fa0/1 are green.

Enter the Switch's CLI and administratively shut down interface Fa0/1:

Switch> enable
Switch# configure terminal
Switch(config)# interface FastEthernet0/1
Switch(config-if)# shutdown
Switch(config-if)# end

Now look at the Packet Tracer topology. You will see the link light on Switch port Fa0/1 turn red. The link light on the PC side will also usually turn red.

PT Visual: The link light on Switch port Fa0/1 is red. The link light on the PC side is also red.

Check the interface status on the switch:

Switch# show ip interface brief

PT Visual: The output of show ip interface brief for FastEthernet0/1 shows "administratively down" in the "Status" column and "down" in the "Protocol" column.

Try to ping the switch's management IP (if configured) or another device on the same network from the PC. It will fail.

Solution Steps:

Enter the Switch's CLI.

Enter the configuration mode for the shutdown interface Fa0/1.

Apply the no shutdown command to enable the interface:

Switch# configure terminal
Switch(config)# interface FastEthernet0/1
Switch(config-if)# no shutdown
Switch(config-if)# end

Verifying the Solution:

Look at the Packet Tracer topology. You will see the link lights on Switch port Fa0/1 and on the PC turn green (after a short delay for STP).

PT Visual: The link lights on Switch port Fa0/1 and on the PC are green.

Check the interface status on the switch again:

Switch# show ip interface brief

PT Visual: The output of show ip interface brief for FastEthernet0/1 now shows "up" in the "Status" column and "up" in the "Protocol" column.

Try to ping the switch's management IP or another device on the same network from the PC. It will be successful.
